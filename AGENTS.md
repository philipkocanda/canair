For reference, the WiCAN firmware is checked out in the `wican-fw/` directory (gitignored; pull if you need to reference the latest version).

## Running the CLI ŌĆö ALWAYS use `uv run canair` from the repo root

**Every `canair` invocation in this repo MUST be run as `uv run canair ŌĆ”` from the project root (`/Users/philip/projects/canair`).** Do NOT call a bare `canair` (a globally installed `uv tool install .` copy). Running bare `canair`:

- may execute **stale code** from a previous install instead of the current working tree, and
- may resolve a **different vehicle profile** (wrong `--profiles-dir` / `default_profile`) than the repo-bundled `profiles/ioniq-2017/`.

Running `uv run canair ŌĆ”` from the repo root guarantees the latest repo code and the expected in-repo profile. Throughout this file and the skills, wherever an example is written as `canair ŌĆ”`, execute it as `uv run canair ŌĆ”` (use the `workdir` of the repo root; never `cd`-hop away from it).

## Prefer canair's built-in tooling ŌĆö do not hand-roll

For **all** querying, capturing, analysis, and reverse-engineering, use the dedicated `canair` subcommands documented below rather than raw sockets, ad-hoc scripts, or hand-editing YAML:

- **Query/read the device** with `canair query` / `scan` / `discover` / `io` / `routines` (use `raw` only as a last resort).
- **Analyze historical data** with `canair captures`, `canair decode`, `canair coverage`, `canair research`.
- **Edit definitions** with `canair pids` (surgical, validated) ŌĆö not by hand-editing `ecus/` YAML.
- **Always pass `--save`** (with `--label`, `--state`, `--notes`) when reading from the device so every payload is recorded to `captures/`. See **Key Files ŌåÆ captures** below for the full flag set and examples. `--save` works with `query`/`scan`/`raw`/`discover`, positional query/raw steps, and `--monitor`. Saves are **journaled** (a write-ahead log under `captures/.journal/`) and reconciled into the dated capture file on exit, so a killed/crashed/disconnected session is never lost ŌĆö recover leftovers with **`canair captures --recover`** (`--discard` to drop them). In `--monitor` press **`s`** to set/edit the label/state/notes live; the **vehicle state is auto-suggested** from decoded PID values via the profile's `states.yaml` (see `canair validate states`).

## Tools

> Reverse-engineering a new PID/DID end-to-end (discover ŌåÆ capture ŌåÆ analyze ŌåÆ define ŌåÆ verify) is documented in the **reverse-engineer-pid** skill; general project/device context is in the **ioniq-reverse-engineering** skill.

All functionality is exposed through a single installable CLI, **`canair`** ŌĆö but in this repo you **always** invoke it as **`uv run canair ŌĆ”` from the repo root** (see "Running the CLI" above). `uv tool install .` exists for end users, not for agents working in the tree.

- **`canair wican`** ŌĆö Generate WiCAN vehicle profiles from the active profile's `ecus/`, upload/download/diff with device
- **`canair query`** ŌĆö Custom CAN/UDS requests over the configured transport (raw `slcan-tcp` by default, or the `wican-ws` WebSocket ELM327 terminal). **Prefer positional query steps** like `canair query BMS:2101` (multi mini-language ŌĆö handles sessions, wake, keepalives; a bare selector like `BMS:2101` is treated as a query step). **Bind each PID to its ECU with a colon (`IGPM:22BC07`), never a space** ŌĆö in a query step a space separates *independent ECU selectors*, so `"query IGPM 22BC07"` means "all of IGPM plus a bogus ECU 22BC07" and is rejected. Use `canair query --param`/`canair query BMS` for simple single-ECU reads, `canair scan range` for discovery. Companions: `canair discover`, `canair io`, `canair routines`, `canair identity` (standard UDS identity DIDs ŌĆö F100/F18x/F190/F19x), `canair scan routines` (probe requestRoutineResults across a range ŌĆö auto-selects UDS 0x31 SF03 or KWP2000 0x33 RequestRoutineResultsByLocalIdentifier by `id_protocol`, so it never blind-sends 0x31 StartRoutine to a KWP2000 ECU; hits saved to `ecus/<ecu>.yaml` `routines:`), `canair scan iocontrol` (probe returnControlToECU across a DID/LID range ŌĆö auto-selects UDS 0x2F or KWP2000 0x30 by `id_protocol`; hits saved to `iocontrol_discoveries:`), `canair tester-present`, `canair raw` (**last resort** ŌĆö hex dump only, no decoding), `canair repl` (interactive). `--verbose` is for debugging canair itself, not normal use. **Use `--reboot` to restore AutoPID after session** (WebSocket terminal overrides AutoPID mode). Dependencies: `websockets`, `pyyaml`, `requests` (optional, for reboot).
- **`canair captures`** ŌĆö Query captured UDS payloads across all capture files. Use after adding new captures to spot patterns. Takes a QUERY (mini-language, like the other tools): `BMS 2102` (ECU + PID, most useful), `"BMS:2102,2103"` (several PIDs), `BMS` (all PIDs for an ECU), or a quoted cross-ECU `"VCU:2101 BMS:2101"`. Add `--diff` for the byte-level diff view or `--step` to interactively step through captures. Standalone modes (no QUERY): `--summary` (stats per ECU/date), `--sessions`/`-n` (session **table of contents** ŌĆö date/time-span/state/label/session-notes/ECUs + distinct capture-notes per session, no payloads; the fast "what's in the captures?" index, `--json` for machine output), `--latest [ECU]` (most recent payload per PID), `--recover` (reconcile orphaned journals from a killed/crashed `--save` session into capture files; `--discard` to delete them unsaved). Any mode can be scoped to a date range with `--since`/`--until` (inclusive, `YYYY-MM-DD`) or `--date` (single day), and/or by `--state SUBSTR`/`--label SUBSTR` (case-insensitive substring of the session state/label).
- **`canair decode`** ŌĆö Parameter/value-centric decoding: applies WiCAN expressions to all historical captures. **Default `canair decode BMS 2101` shows each param's value range** across captures (payload/byte-diff views live in `canair captures`). Modes: `--param` (filter), `--compact` (one aligned row per capture ŌĆö param names in a single header, `[state]` divider only when the session state changes, time-only rows, right-aligned value columns; add `--changes-only`/`-c` to drop rows unchanged from the previous one, collapsing stationary runs), `--stats` (n/distinct/mean/median/stdev per param; add `--group-by state` for per-drive-segment stats in one shot), `--corr PARAM` (Pearson correlation of every param vs a reference ŌĆö validate a candidate against a known signal), `--plot` (interactive signal explorer: sweep ImHex-style byte interpretations `u8/i16/f32/ŌĆ”`├Śendianness and params, plot across captures, apply transforms `delta/abs/normalize/ŌĆ”`, zoom/pan the x-axis, overlay a `--corr` signal, show the visible date/time range, list the backing captures in a modal (`i`), flag bytes already mapped by a defined param, and read off the equivalent WiCAN expression), `--try "NAME[:unit]=EXPR"` (test a candidate expression against captures **without** editing YAML; works even for a not-yet-defined PID), `--verified`/`--unverified`, `--json`. **Scope which captures are considered** (shared with `canair captures`): `--since`/`--until`/`--date` (inclusive `YYYY-MM-DD`), `--state SUBSTR`/`--label SUBSTR` (case-insensitive substring of the session state/label ŌĆö the natural unit of drive analysis, e.g. `--state 'MT->KW'`), and `--first N`/`--last N` (chronological slice). Example drive analysis: `canair decode MCU 2102 --stats --group-by state --state driving`.
- **`canair research`** ŌĆö Report the open reverse-engineering backlog from the per-ECU `research:` sections in the profile's `ecus/`. Complements `canair coverage` (undecoded *bytes*) by surfacing *planned* work (scans/decodes/verifies). Modes: `canair research` (all open, priority-sorted), `--summary` (counts by status/type/priority/ECU), `--ecu`, `--type scan|decode|verify|iocontrol_scan`, `--status`, `--priority P1|P2|P3`, `--prerequisite acc|ready|ŌĆ”`, `--all` (include done), `--json`. Use as the "what should I RE next?" entry point.
- **`canair coverage`** ŌĆö Audit PID definitions for decoding gaps. Cross-references each PID's parameter expressions against its longest captured payload and reports **UNMAPPED** data bytes, incomplete **BITS** (bytes read bit-by-bit with undecoded bits left), and **NO CAPTURE** PIDs (params defined but nothing captured). Modes: `canair coverage` (all), `canair coverage IGPM [22BC03]` (filter), `--bitfields`, `--unmapped`, `--no-capture`, `--all` (include fully-mapped), `--json`. Run after adding params/captures to spot what still needs decoding.
- **`canair pids`** ŌĆö Safely add/update the profile's `ecus/` **parameters** and **research** entries from the CLI (surgical, comment-preserving; each edit is YAML-reparsed and schema-validated via `canair validate pids`, and auto-reverted on failure). Subcommands: `upsert-param ECU PID NAME EXPR [--unit --min --max --source --notes --verified/--unverified ŌĆ”]`, `add-research ECU --type --target --status [--priority --prereq ŌĆ”]`, `set-status ECU TARGET STATUS [--type]`. Prefer this over hand-editing the rich per-ECU YAML.
- **`canair config`** ŌĆö View and manage user config (`~/.config/canair/config.yaml`): transport block, `wican_addresses`, `default_wican`, `default_profile`, `profiles_dir`. Subcommands: `show` (default ŌĆö config file locations + effective settings + resolved transport + WiCAN address book; `--json`), `path` (print user config file path), `get KEY`, `set KEY VALUE` (dotted keys create nested mappings, e.g. `transport.type`, `wican_addresses.home`; int/bool coerced unless `--string`), `unset KEY`, `edit` (`$EDITOR`). Comment-preserving edits. Prefer this over hand-editing `config.yaml`.
- **`canair dtc`** ŌĆö Read stored Diagnostic Trouble Codes (DTCs) for one ECU or `--all` (UDS `0x19` / KWP2000 `0x18`, auto-selected by `id_protocol`). Logs each scan to the profile's `dtc_log.yaml` and reports what **changed** since the last scan (DTC *meanings* live per-ECU in each `ecus/<name>.yaml` `dtcs:` section; profile-wide `failure_types:` in `profile.yaml`) (DTC *meanings* live per-ECU in each `ecus/<name>.yaml` `dtcs:` section; profile-wide `failure_types:` in `profile.yaml`); clear fault memory with `--clear` (`0x14`). Read-only unless `--clear` is passed.
- **`canair sniff`** ŌĆö Passive CAN-bus sniffer (raw `slcan-tcp` backend **only** ŌĆö asks to switch the device to `slcan` if needed). Live per-ID table (count/rate/last data/changed bytes) for discovering broadcast IDs the request/response path can't see; optional `.asc`/`.blf`/`.csv` frame logging via `--save`, plus `--filter`/`--listen-only`/`--duration`. Complements the active request/response tools which can't observe passive broadcasts.
- **`canair status`** ŌĆö Read-only snapshot of the configured transport, device mode, and reachability: "what am I talking to, in what mode, is it usable?". Use before a session to confirm the device is in the expected protocol (`slcan` vs `auto_pid`).
- **`canair ecu`** ŌĆö Inspect the profile's ECU registry (aggregated from the per-ECU `ecus/` files) and per-ECU stats (PID/param/verified counts, protocol, addresses). Companion to `canair validate ecus`.

### Profiles

Vehicle data lives in a **profile** bundle ŌĆö a directory with `ecus/` (one file per ECU ŌĆö the source of truth), `profile.yaml` (profile-wide settings), `captures/`, `references/`, `states.yaml`, and generated `out/`. The repo ships `profiles/ioniq-2017/` as the default/example. Selection precedence: `--profile NAME|PATH` (global flag, before the subcommand) > `CANAIR_PROFILE` env var > `default_profile` in config > single discovered profile (auto). Profiles are discovered from `--profiles-dir`, `$CANAIR_PROFILES_DIR`, `profiles_dir` in config, `~/.config/canair/profiles/` (user, uncommitted), and the repo's bundled `profiles/`; user profiles shadow bundled ones by name. Inspect with `canair profile list` / `show [NAME]` / `path [NAME]`. Scaffold a new vehicle with `canair profile create <name> --car-model "ŌĆ”"` (creates `profile.yaml`, empty `ecus/`, `captures/`, `out/` under `~/.config/canair/profiles/<name>` or `--path DIR`; `--set-default` to adopt it).

## Key Files

- **`profiles/ioniq-2017/ecus/`** ŌĆö SOURCE OF TRUTH, one file per ECU. Each `ecus/<name>.yaml` fully describes one ECU: `identity:` (name is the file's top key), `scan_log:` (probe history), `dtcs:` (DTC meanings), `pids:`, `iocontrol:`, `routines:`, `research:`. Identity-only modules (AMP/SRS) have no `pids:`. Validate with `canair validate pids` (alias `ecus`); `--stats` for counts
- **`canair validate`** ŌĆö Schema validation with a positional target: `canair validate pids|captures|ecus|states|all` (default `all`). Add `--stats` for a count summary (ECUs/PIDs/params/verified/research items). Schemas are tool-owned in `canlib/schema/` (`pids_schema.yaml` ŌĆö covers the per-ECU files incl. identity/scan_log/dtcs, `captures_schema.json`, `states_schema.yaml`). `validate ecus` is an alias for `validate pids`. `validate captures` also emits **soft warnings** for session `state`s outside the profile's `states.yaml` vocabulary (free text still accepted).
- **`profiles/ioniq-2017/captures/`** ŌĆö Raw UDS response payloads, split by date (e.g. `2026-04-19.yaml`). Schema in `captures/SCHEMA.yaml` (tool-owned: `canlib/schema/captures_schema.json`). Validate with `canair validate captures`. **NEVER hand-write or edit these YAML files.** Record device reads via `canair query ŌĆ” --save` with `--label`/`--state`/`--notes` (non-interactive), e.g. `canair query "query MCU" "query VCU:2101" --wican vpn --save --label "ŌĆ”" --state "ready, parked" --notes "ŌĆ”"`. `--save` works with `canair scan`/`raw`/`discover`, positional query/raw steps, and `--monitor`. Saves are **journaled** to `captures/.journal/` as they stream and reconciled on exit; recover a killed session's leftovers with `canair captures --recover`. For edits/removals use `canlib.captures` helpers (`set_capture_note`, `delete_capture`). **After adding captures, run `canair captures --summary` to check for new patterns.**
- **`profiles/ioniq-2017/states.yaml`** ŌĆö Canonical, ordered vehicle operating states (`sleep/plugged/acc/acc2/ready/charging` + composites), each with an optional `when:` predicate over decoded `ECU.PARAM` values used to **auto-suggest** the capture `state` (see `canlib/states.py`). Schema `canlib/schema/states_schema.yaml`; validate with `canair validate states`. `canair validate captures` soft-warns on session states with no token in this vocabulary. New profiles are scaffolded with a starter `states.yaml`.
- **`profiles/ioniq-2017/references/`** ŌĆö External reference material (other-vehicle logs, spreadsheets, SavvyCAN exports, charge-curve data) used to cross-reference RE work. Distinct from the per-ECU `research:` backlog sections inside `ecus/` (surfaced by `canair research`).
- **`profiles/ioniq-2017/profile.yaml`** ŌĆö Profile-wide settings: `car_model`, `init`, `response_timeout_ms`, `multi_did_batching`, and DTC `failure_types:`.
- **`canair pids`** edits the `ecus/` files (parameters + research); ECU identity/scan_log are written by `canair discover --register` / `canair identity`.
- (There is no longer a top-level `ecus.yaml` or `dtc.yaml` ŌĆö both were folded into the per-ECU `ecus/` files.)
- **`canair bix`** ŌĆö Byte index converter: WiCAN Ōåö ISO-TP Ōåö Torque Ōåö bix. Use `canair bix w9` or `canair bix E` for quick lookups, `--table` for full table. **`--annotate HEX` (`-a`)** maps a raw UDS response payload to a table with WiCAN Bnn, ISO-TP index, Torque letter, bix, and role per byte. Supports `-1` (21xx, default) and `-2` (22xxxx) subfunction modes.
- **`docs/wican-iso-tp-index-conversion.md`** ŌĆö Reference table for byte index notation differences (local only, not tracked in git)
- **`docs/CLI commands.md`** ŌĆö Reference for `canair query` usage and examples (local only, not tracked in git)

## WiCAN Access

Device addresses are configured in `~/.config/canair/config.yaml` (a legacy repo-root `config.yaml` is still read for back-compat; both gitignored). Copy from `config.example.yaml`:

```yaml
wican_addresses:
  home: "10.0.2.86"       # Device on local LAN
  vpn: "192.168.3.2"      # Device via VPN
default_wican: home
```

All CLI tools use `--wican home|vpn|<ip>` to select the target. Without `config.yaml`, falls back to `192.168.80.1` (WiCAN AP mode).

- WebSocket terminal: `ws://<ip>/ws` (send `{"ws_mode": "terminal", "terminal_type": "elm327"}`)
